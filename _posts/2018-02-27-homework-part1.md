# Chapter 1을 읽고 성능 테스트의 정의를 정리

### 성능 테스트란?
  - 주로 시스템의 병목을 확인하거나, 미래의 테스트의 베이스라인을 만들거나, 성능 튜닝을 서포트하거나, 성능  골이나 요구사항을 결정하거나, 이해 관계자들이 전체적인 어플리케이션의 품질과 관련해서 결정을 내릴 때에 쓰일 성능 관련 데이타를 수집할 때 수행되는 것.

### 성능 테스트를 구성하는 활동 7가지
  1. 테스트 환경 확인하기
  2. 성능 허용 기준 확인하기
  3. 테스트 계획 및 설계하기
  4. 테스트 환경 설정하기
  5. 설계된 테스트 구현하기
  6. 테스트 수행하기
  7. 결과 분석 및 리포팅, 그리고 재테스트

### 왜 성능 테스트를 하는지?
- 배포 준비 평가
- 개발된 소프트웨어 성능의 적절성 평가
- 성능 튜닝의 효율성 향상

### 성능 테스트와 프로젝트 컨텍스트
- 성능 테스트가 성공할 수 있으려면, 테스트 접근법과 테스트 그 자체가 프로젝트의 컨텍스트와 관련이 있어야 한다.
- 프로젝트 컨텍스트에 대한 이해가 없다면, 성능 테스트는 단지 테스트를 수행하는 주체가 중요하다고 생각하는 요소에 국한된다.
#### **프로젝트 컨텍스트**
  - 프로젝트 성공 달성과 관련된 것들
  - 다음과 같은 것들을 포함한다
  - *프로젝트의 전반적인 비전 또는 의도, 성능 테스트 목표, 성능 성공 기준, 개발 수명 주기, 프로젝트 스케쥴, 프로젝트 예산, 사용 가능한 툴과 환경, 성능 테스트를 수행하는 주체들의 스킬 셋, 발견된 성능 문제의 우선순위, 배포된 어플리케이션이 좋지 않게 수행되었을 때의 비즈니스 파급력*
  - 프로젝트 컨텍스트에서 성능 테스트 작업과 관련있는 항목으로는 들의 예
      - 프로젝트 비전
      - 시스템의 목적
      - 소비자나 사용자의 기대
      - 비즈니스 드라이버 (비즈니스 요구나 기회 등)
      - 성능 테스트의 이유
      - 성능 테스트가 프로젝트에 가져올 가치
      - 프로젝트 관리 및 직원 배치 ~~staffing~~
      - 프로세스
      - 준수 기준 ~~compliance criteria~~
      - 프로젝트 스케줄
### 성능 테스트의 카테고리
  - 퍼포먼스 테스트
  - 로드 테스트
  - 스트레스 테스트
### 기준선과 벤치마킹
#### 기준선
- 성능 측정 항목 데이터를 얻기 위해서 기준선을 생성한다.
- 기준선을 사용할 때의 고려사항
  - 기준선은 시스템이나 컴포넌트, 어플리케이션에 의해 생성될 수 있다.
  - 기준선은 미래의 최적화나 회귀를 추적하기 위해, 비교를 위한 표준으로 설정될 수 있다.
  - 기준선은 성능 변화 확인에 도움을 줄 수 있다.
  - 기준선 자산은 재사용이 가능하다
  - 기준선은 메트릭이다.
  - 기준선은 레퍼런스의 공유된 프레임과 같다?
  - 기준선을 과-일반화하지 마라
  - 어플리케이션의 행동을 알아라
  - 기준선은 진화한다.

#### 벤치마킹
- 벤치마킹은 기준선과 시스템의 성능을 비교하는 과정이다.
- 벤치마킹과 관련된 고려사항
  - 규칙에 따라서 벤치마크를 진행할 것
  - 규칙에 따라 벤치마크를 진행하기 때문에 투명하게 할 수 있다.
  - 다양한 측정 항목을 통해 결과를 공개한다.

### 그래서 다시 성능 테스트란?
  - 성능 테스트는 시스템의 병목을 확인하고, 미래의 테스트를 위한 기준선을 설립하고, 성능 튜닝 작업을 지원하고, 성능 목표와 요구사항 준수를 결정하는데 도움이 된다.
  - 성공적으로 성능 테스트를 하기 위해서, 테스트는 반드시 정말로 중요한 항목에 집중할 수 있게 하는 **프로젝트 컨텍스트** 와 관련이 있어야 한다.
  - 만약 성능 특성을 용납할 수 없는 경우, 어플리케이션이 수용할 수 있도록 수행되기 위해서 성능 튜닝을 진행할 수 있다.
  또한 미래의 시스템 성능을 향상하고 사용되는 자원의 양을 줄이기 위해서도 튜닝을 할 수 있다.
  - 퍼포먼스, 로드, 스트레스 테스트는 성능 테스트의 하위 카테고리이다.
  - 나중의 성능 향상 변화의 효율을 평가하기 위해 기준선을 만드는 것은 프로젝트 효율을 향상시킬 것이다.

# Chapter 2를 읽고 성능 테스트의 종류를 정리 & 해당 테스트가 어떤 상황에서 필요한지 생각하여 정리 (word 2 page)

#### 퍼포먼스 테스트
- 목적
  - 어플리케이션의 속도, 확장성, 안정성을 결정하기 위해서 수행된다.
- 언제 필요한지
  - 어플리케이션이 사용자의 요구를 만족하고 있는지 검증해야 할 때
- 특징
  - 부하가 적을 때 나타나는 문제는 못 잡을 수 있다.

#### 부하 테스트
- 목적
  - 부하가 정상적일 때와 부하가 심할 때 어플리케이션의 동작을 검증하기 위해서 수행된다.
- 언제 필요한지
  - 동시성 이슈를 발견할 때
  - 부하 하에서 기능상의 에러를 찾을 때
- 특징
  - 부하 테스트의 목표는 Service Level Agreement (SLA) 를 만족시키는 것이다.
  - Endurance testing
    - 부하 테스트의 부분집합
    - Mean Time Between Failure (MTBF), Mean Time To Failure (MTTF) 등을 계산
  - 응답시간이 주 목적인 테스트가 아니다

####  스트레스 테스트
- 목적
  - 부하가 엄청 많이 생겼을 때 어플리케이션의 동작을 검증하기 위해서 수행된다.
- 언제 필요한지
  - 어플리케이션에 많은 부하가 가해졌을 때 동기화 이슈, race condition, 메모리 누수 등 부하가 높을 때의 버그를 발견하기 위한 테스트
- 특징
  - Spike testing
    - 짧은 시간동안 반복적으로 많은 양의 작업량이 주어질 때에 대한 테스트?

#### 수용량 테스트
- 목적
  - 시스템이 꾸쭌히 성능 목표를 만족하면서 얼마나 많은 양의 사용자나 트랜잭션을 수행할 수 있는지 결정하기 위해서 수행된다.
- 언제 필요한지
  - 스케일 업이나 스케일 아웃을 하기 전에
- 특징
  - 카파시티 테스트는 카파시티 플래닝과 함께 쓰인다
  - 예를 들어 미래의 로드에 대처하기 위해 프로세서, 메모리, 디스크, 네트워크 등 얼마나 많은 양의 리소스가 필요한지 알아야 할 때 등

#### 기타 다른 테스트
- Component test, investigation, smoke test, unit test, validation test와 같은 테스트들이 있다.

#### 요약
- 성능 테스트 과정에서 위험을 줄이고 비용을 최소화하며 적절한 테스트를 적용해야하는 시기를 파악하기 위해서는 다양한 성능 테스트 유형을 이해하는 것이 중요하다.
성능 테스트를 통해 다양한 테스트 유형을 적용하려면 다음과 같은 사항을 평가해야한다.
  - 성능 테스트의 목적
  - 성능 테스트의 컨텍스트
    - 관련 리소스, 비용 및 테스트 노력에 대한 잠재적 수익 등

# Chapter 3 을 읽고 성능 테스트시에 어떤 위험사항이 있는지 정리

### 성능 테스트 타입에 따른 위험들

#### 수용량 테스트
- 시스템 수용량이 부하 하에서 비즈니스가 동작할 수 있는 용량을 만족하는지?

#### 컴포넌트 테스트
- 해당 컴포넌트가 예상대로 동작하는지?
- 해당 컴포넌트가 잘 최적화되었는지?
- 발견되는 성능 이슈가 해당 컴포넌트로 인해 생겨났는지?

#### 인내 테스트
- 시간이 지나도 성능이 일관되는지?
- 발견되지 않은 '느리게 성장하는 문제'가 있는지?
- 만나지 못한 외부 간섭이 있는지?

#### 조사 테스트
- 시간에 따른 성능 트렌드는 어떻게 되는지?
- 미래의 테스트를 위해서 어떤 것을 비교해야 하는지?

#### 부하 테스트
- 얼마나 많은 유저들을 어플리케이션에서 다룰 수 있는지?
- 얼마나 많은 데이터들을 현재의 데이터베이스 서버/파일 서버에서 다룰 수 있는지?
- 적절한 네트워크 컴포넌트가 있는지?

#### 연기 테스트
- 추가 성능 테스트를 위해 해당 빌드/설정을 사용할 준비가 되었는지?
- 다음에 어떤 성능 테스트를 하면 되는지?
- 해당 빌드가 지난 번보다 좋은 성능 / 나쁜 성능을 보이는지?

#### 스파이크 테스트
- 프로덕션 부하가 예상한 최대 부하보다 높으면 어떤 일이 일어나는지?
- 어떤 종류의 실패를 계획해야 하는지?
- 어떤 지표를 봐야 하는지?

#### 스트레스 테스트
- 프로덕션 부하가 예상 부하보다 높으면 어떤 일이 일어나는지?
- 어떤 종류의 실패를 계획해야 하는지?
- 실패 전에 개입하기 위해 어떤 지표를 봐야 하는지?

#### 단위 테스트
- 해당 코드 뭉치가 효율적인지?
- 성능 예산 안에 있는지? ~~??~~
- 해당 코드가 기대 부하 하에서 동작하는지? ?????

#### 검증 테스트
- 어플리케이션이 목표와 요구사항을 만족하는지?
- 해당 버전이 전보다 느리거나 빠른지?
- 릴리즈했을 때 SLA를 위반하지는 않는지?

### 위험 카테고리
#### 속도 관련 위험
- 사용자 만족
- 동기화
- SLA 위반
- 응답 시간 트렌드
- 설정
- 일관성

#### 확장성 관련 위험
- 수용량
- 용량
- SLA 위반
- 최적화
- 효율성
- 미래 성장
- 자원 소비
- 하드웨어 / 환경

#### 안정성 관련 위험
- 신뢰성
- 견고성
- 하드웨어 / 환경
- 실패 모드?
- 느린 누수
- 복구
- 데이터 정확성 및 보안
- 인터페이스

### 요약
- 거의 모든 응용 프로그램 및 비즈니스 관련 위험은 사용자 만족도 및 응용 프로그램의 비즈니스 목표 달성 능력을 포함하여 성능 테스트를 통해 해결할 수 있다.
- 일반적으로 성능 테스트에서 다루는 위험은 속도, 확장 성 및 안정성 측면에서 분류됩니다. 속도는 일반적으로 최종 사용자의 우려 사항이며 확장성은 비즈니스 관심사이며 안정성은 기술 또는 유지 관리 문제이다.
- 프로젝트 관련 위험 및 성능 테스트를 사용할 수 있는 관련 완화 전략을 식별하는 것은 거의 보편적으로 가치 있고 시간을 절약하는 방법으로 간주된다.
